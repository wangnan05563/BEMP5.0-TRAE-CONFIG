# 性能测试最佳实践

## 1. 测试策略

### 1.1 渐进式压测

| 阶段 | 并发数 | 持续时间 | 目标 |
|------|--------|----------|------|
| 基准测试 | 10 | 1 分钟 | 验证系统基本功能 |
| 负载测试 | 50 | 5 分钟 | 确定系统正常负载能力 |
| 压力测试 | 100 | 10 分钟 | 确定系统最大处理能力 |
| 极限测试 | 200+ | 15 分钟 | 寻找系统崩溃点 |

### 1.2 测试类型

| 类型 | 描述 | 适用场景 |
|------|------|----------|
| 负载测试 | 模拟正常用户负载 | 日常性能评估 |
| 压力测试 | 超出正常负载的压力 | 确定系统极限 |
| 稳定性测试 | 长时间持续运行 | 验证系统稳定性 |
| 并发测试 | 大量用户同时操作 | 验证并发处理能力 |

## 2. 测试环境要求

### 2.1 环境隔离

- **强制**: 压测必须在独立测试环境执行
- **推荐**: 测试环境配置应与生产环境相似
- **参考**: 使用容器化技术快速搭建测试环境

### 2.2 数据准备

- **强制**: 准备充足的测试数据（大于预期请求数）
- **推荐**: 使用参数化数据避免缓存影响
- **参考**: 数据应覆盖各种边界情况

### 2.3 资源监控

| 资源 | 监控工具 | 告警阈值 |
|------|----------|----------|
| CPU | top, htop | >80% |
| 内存 | free, vmstat | >85% |
| 磁盘 I/O | iostat | >80% |
| 网络 | iftop, nethogs | >80% 带宽 |
| 数据库 | slow_query_log | 慢查询 >1s |

## 3. 脚本设计规范

### 3.1 测试计划组织

```
Test Plan
├── 用户定义的变量
├── HTTP 请求默认值
├── HTTP 头信息管理器
├── Cookie 管理器
├── 线程组
│   ├── 事务控制器
│   │   └── HTTP 请求
│   │       ├── 响应断言
│   │       └── JSON 提取器
│   ├── 汇总报告
│   └── 聚合报告
└── 查看结果树（调试用，压测时禁用）
```

### 3.2 参数化方法

| 方法 | 用途 | 示例 |
|------|------|------|
| 用户自定义变量 | 全局配置 | `${host}`, `${port}` |
| CSV Data Set Config | 大批量测试数据 | `${username}`, `${password}` |
| 函数助手 | 动态数据生成 | `${__Random(1,100)}` |
| 正则表达式提取器 | 从响应提取动态值 | `${token}` |
| JSON Extractor | 从 JSON 提取数据 | `${$.data.id}` |

### 3.3 断言规则

**强制配置**:
1. 响应码断言（200）
2. 响应时间断言（<5000ms）
3. 关键业务字段断言

**推荐配置**:
1. 响应内容包含断言
2. JSON 结构验证断言
3. 数据库验证断言

## 4. 结果分析方法

### 4.1 关键指标解读

| 指标 | 优秀 | 良好 | 一般 | 较差 |
|------|------|------|------|------|
| TPS | >1000 | 500-1000 | 100-500 | <100 |
| 平均 RT | <200ms | 200-500ms | 500-1000ms | >1000ms |
| P95 RT | <500ms | 500-1000ms | 1000-2000ms | >2000ms |
| 错误率 | <1% | 1-5% | 5-10% | >10% |

### 4.2 瓶颈识别方法

1. **应用层瓶颈**: 代码逻辑复杂、数据库查询慢、缓存未命中
2. **中间件瓶颈**: 连接池不足、线程池满、队列积压
3. **系统层瓶颈**: CPU 瓶颈、内存不足、磁盘 I/O 瓶颈
4. **数据库瓶颈**: 慢查询、索引缺失、锁竞争

### 4.3 优化建议分类

| 类别 | 优化方向 | 示例 |
|------|----------|------|
| 代码优化 | 算法优化、缓存使用 | 使用 Redis 缓存热点数据 |
| 数据库优化 | 索引优化、查询优化 | 添加联合索引 |
| 架构优化 | 负载均衡、读写分离 | 使用 Nginx 负载均衡 |
| 配置优化 | 连接池、线程池 | 调整 Tomcat 最大线程数 |

## 5. 报告编写规范

### 5.1 报告结构

1. **测试概述**: 目标、时间、环境
2. **测试配置**: 并发数、持续时间、循环次数
3. **关键指标**: TPS、响应时间、错误率
4. **趋势分析**: 图表展示
5. **瓶颈分析**: 已识别的问题
6. **优化建议**: 具体的改进措施
7. **结论**: 总体评估

### 5.2 图表要求

- **强制**: TPS 趋势图
- **强制**: 响应时间分布图
- **推荐**: 错误率饼图
- **推荐**: 资源监控仪表盘

## 6. 常见问题与解决方案

### Q1: 压测结果不稳定

**可能原因**:
- 测试数据不足导致缓存命中率高
- 其他系统干扰（如定时任务）
- 网络波动

**解决方案**:
- 使用参数化数据
- 在独立环境测试
- 多次测试取平均值

### Q2: 压测时系统资源未打满但 TPS 上不去

**可能原因**:
- 应用层代码性能问题
- 数据库查询慢
- 外部依赖响应慢

**解决方案**:
- 使用 APM 工具定位瓶颈
- 优化慢查询 SQL
- 增加缓存层

### Q3: 分布式压测结果不准确

**可能原因**:
- Master 和 Slave 时钟不同步
- 网络延迟影响
- Slave 机器性能差异

**解决方案**:
- 使用 NTP 同步时钟
- 选择网络稳定的环境
- 使用配置相同的 Slave 机器
