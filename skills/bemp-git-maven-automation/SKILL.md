# Git Maven 自动化构建技能

该技能自动化执行以下流程：
1. 智能发现工作空间中的所有Git版本库，排除指定目录
2. 生成详细的代码拉取清单，包含仓库状态、分支信息、远程URL等
3. 对清单中的每个仓库依次执行 `git fetch` 和 `git pull` 操作，支持冲突检测和处理
4. 所有仓库同步成功后，根据配置执行全量构建或增量构建
5. 构建完成后生成详细的构建报告，包含构建时间、成功率等信息

## 功能特点

- **智能仓库发现**: 自动检测工作空间中的所有Git仓库，支持排除指定目录，避免扫描无关仓库
- **详细拉取清单**: 生成包含仓库路径、当前分支、远程URL、上次同步时间、仓库状态等详细信息的拉取清单
- **冲突智能处理**: 检测到冲突时立即停止并提供详细冲突信息，支持自动备份冲突文件
- **构建模式支持**: 支持全量构建和增量构建两种模式，增量构建可指定目标模块
- **构建顺序优化**: 智能分析模块依赖关系，按最优顺序执行构建，提高构建效率
- **并行构建支持**: 支持多模块并行构建，显著减少构建时间
- **错误智能处理**: 完善的错误处理机制，自动识别常见错误并提供解决方案
- **构建报告生成**: 构建完成后生成详细的构建报告，包含构建时间、成功率、模块状态等信息
- **实时进度反馈**: 在IDE内部终端输出详细执行日志和实时进度
- **配置灵活性**: 支持通过环境变量或配置文件进行详细配置，满足不同场景需求

## 使用场景

- 项目包含多个Git子模块或独立仓库需要同步更新
- 团队协作环境下需要定期拉取最新代码并重新构建
- 持续集成/部署前的本地代码同步和构建验证
- 多仓库项目的统一版本管理和构建流程

## 实现原理

1. **智能仓库发现**: 递归扫描工作空间目录，查找包含 `.git` 文件夹的目录，支持排除指定目录（如 `node_modules`、`target` 等）
2. **详细拉取清单生成**: 为每个发现的仓库收集信息，包括仓库路径、当前分支、远程URL、上次同步时间、仓库状态等，生成结构化的拉取清单
3. **Git操作执行**: 使用Trae的Git插件执行 `git fetch` 和 `git pull` 命令，支持自定义Git参数
4. **冲突智能检测**: 检查 `git pull` 命令输出，检测冲突关键字，提供详细的冲突文件列表
5. **构建类型确定**: 根据配置或用户选择，确定执行全量构建或增量构建
6. **构建顺序优化**: 分析模块间的依赖关系，生成最优构建顺序，确保依赖模块先于被依赖模块构建
7. **并行构建执行**: 支持多模块并行构建，通过 `-T` 参数指定线程数，提高构建效率
8. **错误智能处理**: 自动识别常见构建错误，提供详细的错误分析和解决方案
9. **构建报告生成**: 构建完成后，汇总构建结果，生成详细的构建报告
10. **状态全程跟踪**: 记录每个步骤的执行结果和错误信息，提供完整的执行历史

## 注意事项

- 所有Git操作必须通过Trae的Git插件执行，确保在IDE内部终端输出日志
- 检测到冲突时，技能会立即停止，用户需要手动解决冲突后重新执行
- 技能支持无根POM的项目结构，会按依赖顺序构建各个模块
- 确保网络连接正常，能够访问远程Git仓库
- 如果Maven构建过程中被中断，需要使用 `StopCommand` 工具终止进程后重新执行
- 并行构建可能会增加系统资源消耗，建议根据系统配置调整线程数
- 增量构建时，确保指定的目标模块依赖已经构建完成
- 首次构建时，建议执行全量构建，确保所有依赖模块都正确构建
- 定期清理Maven本地仓库，避免依赖冲突和构建失败

## 错误处理

技能包含以下智能错误处理机制：
- **Git命令错误处理**: Git命令执行失败时记录详细错误信息，支持重试机制和错误分类
- **仓库同步错误处理**: 仓库同步失败时可配置跳过该仓库并继续处理其他仓库，或停止整个流程
- **网络连接错误处理**: 网络连接超时或远程仓库不可达时，自动检测网络状态并提供跳过选项
- **冲突处理机制**: 检测到代码冲突时，立即停止并提供详细的冲突文件列表，支持自动备份冲突文件
- **Maven构建错误处理**: 构建失败时自动分析错误类型，提供详细的错误信息和针对性的解决建议
- **构建中断处理**: 构建过程被中断时，自动使用 `StopCommand` 工具终止进程并清理临时文件
- **测试失败处理**: 测试用例失败时，自动使用 `-DskipTests=true` 跳过测试并继续构建
- **依赖冲突处理**: 检测到依赖冲突时，提供依赖树分析和冲突解决建议
- **内存不足处理**: 构建过程中出现内存不足时，自动调整Maven内存参数并重新尝试
- **所有错误信息都会在IDE终端中高亮显示，提供清晰的错误分类和解决建议**

## 配置选项

技能支持以下配置（通过环境变量或配置文件）：
- `SKIP_REPOS`: 跳过的仓库列表（逗号分隔）
- `SKIP_DIRS`: 跳过的目录列表（逗号分隔），如 `node_modules,target,.idea`
- `MAVEN_OPTS`: Maven构建的额外参数
- `CONFLICT_ACTION`: 冲突处理方式（stop/warn/skip）
- `BUILD_TYPE`: 构建类型（full/incremental），默认为incremental
- `BUILD_TARGETS`: 增量构建的目标模块列表（逗号分隔）
- `PARALLEL_BUILD`: 是否启用并行构建（true/false），默认为false
- `BUILD_THREADS`: 并行构建的线程数，默认为4
- `GIT_RETRY_COUNT`: Git命令失败的重试次数，默认为3
- `GIT_TIMEOUT`: Git命令的超时时间（秒），默认为60
- `Maven_TIMEOUT`: Maven构建的超时时间（分钟），默认为60
- `ENABLE_BUILD_REPORT`: 是否生成构建报告（true/false），默认为true

## 执行步骤

当调用此技能时，请按以下步骤操作：

### 1. 准备工作
- 确认当前工作空间路径：`d:\code\QJ\BEMP5.0DEV`
- 检查网络连接，确保可以访问远程Git仓库
- 确认Maven已安装并配置正确
- 加载并解析配置选项，如 `SKIP_DIRS`、`PARALLEL_BUILD` 等

### 2. 智能发现Git仓库
- 使用 `RunCommand` 工具执行 `Get-ChildItem -Recurse -Directory -Hidden -Filter .git | Select-Object -ExpandProperty Parent | Select-Object -ExpandProperty FullName` 命令（Windows PowerShell）或 `find . -type d -name .git` 命令（Linux/Mac）查找所有Git仓库
- 根据 `SKIP_DIRS` 配置排除指定目录
- 解析命令输出，获取每个仓库的根目录

### 3. 生成详细拉取清单
- 对每个发现的仓库执行以下操作：
  - 执行 `git branch --show-current` 命令获取当前分支
  - 执行 `git remote -v` 命令获取远程URL
  - 执行 `git log -1 --format="%ai" 2>$null` 命令获取上次提交时间，处理无提交记录的情况
  - 执行 `git status --porcelain` 命令获取仓库状态
- **优化清单显示**：
  - 清除终端屏幕，确保清单显示在最上方
  - 使用醒目的标题和分隔线
  - 表格格式显示，包含序号、仓库路径、当前分支、远程URL、上次同步时间、状态等信息
  - 对无提交记录的仓库显示"无提交记录"，避免错误信息
  - 显示仓库总数和状态统计
- 显示清单给用户确认，等待用户选择是否继续

### 4. 同步Git仓库
对于每个发现的Git仓库，按顺序执行以下操作：

#### a. 执行 git fetch
- 使用 `RunCommand` 工具在仓库目录中执行 `git fetch --all` 命令
- 检查命令退出码，如果非零则记录错误并根据配置决定是否重试
- 在IDE终端中输出执行结果，包括获取的新分支和标签

#### b. 执行 git pull
- **记录拉取前的提交哈希**：执行 `git rev-parse HEAD` 命令获取当前提交哈希
- 使用 `RunCommand` 工具在仓库目录中执行 `git pull` 命令
- 分析命令输出，检测是否包含冲突关键字（如 "CONFLICT", "冲突", "merge conflict"）
- 如果检测到冲突，立即停止所有后续操作，向用户报告冲突详情，并提示手动解决
- 如果命令执行失败（非零退出码），记录错误并根据配置决定是否继续处理其他仓库
- **记录拉取后的提交哈希**：执行 `git rev-parse HEAD` 命令获取拉取后的提交哈希
- **检测代码变更**：比较拉取前后的提交哈希，如果不同则标记为有代码变更
- 记录拉取的新提交数量和变更状态

### 5. 命令执行优化
- **PowerShell命令优化**：
  - 使用 `${i}` 语法避免变量引用错误
  - 将长命令分解为多个小命令，避免控制台缓冲区问题
  - 添加异常捕获机制，确保命令执行的可靠性
  - 使用 `try-catch` 块处理可能的执行错误
- **执行策略**：
  - 每个仓库的操作独立执行，避免单个仓库失败影响整个流程
  - 同步和构建操作分离执行，提高错误定位能力
  - 输出详细的执行日志，便于问题排查

### 6. 执行Maven构建
- **构建决策**：
  - 如果是全量构建（`BUILD_TYPE=full`）：直接执行全量构建，不检查代码变更
  - 如果是增量构建（`BUILD_TYPE=incremental`）：检查是否有任何仓库有代码变更
    - 如果没有代码变更，输出"无代码变更，跳过构建"并结束流程
    - 如果有代码变更，执行增量构建

#### 5.1 全量构建
- 遍历每个仓库目录
- 检查每个仓库是否存在 `pom.xml` 文件
- 对于存在 `pom.xml` 的仓库：
  - **banks仓库特殊处理**：
    - 如果仓库路径包含 `banks` 且设置了 `BANKS_DIRS` 环境变量：
      - 解析 `BANKS_DIRS` 环境变量，获取指定的银行目录列表
      - 对于每个指定的银行目录：
        - 检查该目录是否存在 `pom.xml` 文件
        - 如果存在，在该目录下执行构建，使用 `-am` 参数自动构建依赖模块
        - 例如：在 `ext-hnnxbank` 目录下执行 `mvn clean install -am -DskipTests=true`
    - 否则，执行完整构建
  - **其他仓库**：
    - 检查是否启用并行构建
    - 如果启用并行构建，在仓库目录下执行 `RunCommand` 工具运行 `mvn clean install -DskipTests=true -T <线程数>`
    - 如果未启用并行构建，在仓库目录下执行 `mvn clean install -DskipTests=true`
  - 监控构建过程，实时输出构建进度
  - 如果构建失败，自动分析错误类型并提供详细的解决建议
- 构建成功后，生成详细的构建报告

#### 5.2 增量构建
- **智能目标模块检测**：如果未指定 `BUILD_TARGETS`，自动检测代码变更并确定目标模块
  - 遍历每个仓库目录
  - 执行 `git diff` 命令检测哪些模块有代码变更
  - 分析变更模块的依赖关系，确定需要构建的模块列表
  - 生成最优构建顺序
- **手动指定目标模块**：如果指定了 `BUILD_TARGETS`，使用用户指定的模块列表
- 遍历每个仓库目录
- 检查每个仓库是否存在 `pom.xml` 文件
- 对于存在 `pom.xml` 且有代码变更的仓库：
  - **banks仓库特殊处理**：
    - 如果仓库路径包含 `banks` 且设置了 `BANKS_DIRS` 环境变量：
      - 解析 `BANKS_DIRS` 环境变量，获取指定的银行目录列表
      - 对于每个指定的银行目录：
        - 检查该目录是否存在 `pom.xml` 文件
        - 如果存在，在该目录下执行构建，使用 `-am` 参数自动构建依赖模块
        - 例如：在 `ext-hnnxbank` 目录下执行 `mvn clean install -am -DskipTests=true`
    - 否则，执行完整构建
  - **其他仓库**：
    - 检查是否启用并行构建
    - 如果启用并行构建，在仓库目录下执行 `RunCommand` 工具运行 `mvn clean install -pl <模块列表> -am -DskipTests=true -T <线程数>`
    - 如果未启用并行构建，在仓库目录下执行 `mvn clean install -pl <模块列表> -am -DskipTests=true`
  - 监控构建过程，实时输出构建进度
  - 如果构建失败，自动分析错误类型并提供详细的解决建议
- 构建成功后，生成详细的构建报告

### 6. 生成构建报告
- 汇总构建结果，包括构建时间、成功/失败/跳过的模块数
- 生成详细的构建报告，包含构建类型、配置参数、模块状态等信息
- 在IDE终端中显示构建报告
- 根据配置决定是否将构建报告保存到文件

### 7. 错误处理和状态记录
- 每个步骤执行后，记录执行状态（成功/失败/跳过）
- 遇到错误时，提供清晰的错误信息和解决建议
- 所有操作都在IDE内部终端执行，确保日志完整可追溯
- 构建完成后，清理临时文件和进程
