# 安全操作规范

## 1. 操作分级

### 1.1 操作风险等级

| 等级 | 操作类型 | 示例 | 审批要求 |
|------|----------|------|----------|
| 低风险 | 只读查询 | SELECT查询 | 无需审批 |
| 中风险 | 新增数据 | INSERT菜单/参数 | 开发负责人审批 |
| 高风险 | 修改/删除数据 | UPDATE/DELETE | 开发负责人+DBA审批 |
| 极高风险 | 结构变更 | CREATE/DROP TABLE | 开发负责人+DBA+架构师审批 |

### 1.2 环境操作权限

| 环境 | 低风险 | 中风险 | 高风险 | 极高风险 |
|------|--------|--------|--------|----------|
| 开发 | 允许 | 允许 | 允许 | 允许 |
| 测试 | 允许 | 允许 | 需审批 | 需审批 |
| 生产 | 允许 | 需审批 | 需审批 | 禁止远程 |

### 1.3 数据库类型特殊权限

| 数据库类型 | SELECT | INSERT | UPDATE | DELETE | DDL | 说明 |
|-----------|--------|--------|--------|--------|-----|------|
| Oracle MCP | ✓ | ✗ | ✗ | ✗ | ✗ | 仅支持只读查询 |
| MySQL MCP | ✓ | ✓ | ✓ | ✓ | ✓ | 支持全类型SQL |

**MySQL MCP 安全约束**：
- 虽然MySQL MCP支持DML/DDL，但执行前必须完成预检查
- 高风险和极高风险操作仍需审批
- 生产环境建议关闭MySQL MCP的DDL/DML直接执行能力
- 建议为MySQL MCP配置专用数据库用户，实施最小权限原则

### 1.4 MySQL 用户权限控制

为降低 MySQL MCP 全类型 SQL 带来的安全风险，建议创建专用数据库用户，实施最小权限原则：

**用户类型与权限矩阵**：

| 用户类型 | SELECT | INSERT | UPDATE | DELETE | CREATE | ALTER | DROP | INDEX | 适用环境 |
|---------|--------|--------|--------|--------|--------|-------|------|-------|---------|
| bemp_readonly | ✓ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | ✗ | 生产 |
| bemp_readwrite | ✓ | ✓ | ✓ | ✓ | ✗ | ✗ | ✗ | ✗ | 测试/生产 |
| bemp_full | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | 开发 |

**创建用户SQL模板**：

```sql
-- 只读用户（生产环境推荐）
CREATE USER 'bemp_readonly'@'%' IDENTIFIED BY '{password}';
GRANT SELECT ON bemp_hnnx.* TO 'bemp_readonly'@'%';
FLUSH PRIVILEGES;

-- 读写用户（测试环境推荐）
CREATE USER 'bemp_readwrite'@'%' IDENTIFIED BY '{password}';
GRANT SELECT, INSERT, UPDATE, DELETE ON bemp_hnnx.* TO 'bemp_readwrite'@'%';
FLUSH PRIVILEGES;

-- 完全访问用户（仅开发环境）
CREATE USER 'bemp_full'@'%' IDENTIFIED BY '{password}';
GRANT ALL PRIVILEGES ON bemp_hnnx.* TO 'bemp_full'@'%';
FLUSH PRIVILEGES;
```

**权限配置最佳实践**：

| 实践项 | 说明 |
|--------|------|
| 最小权限原则 | 仅授予完成操作所需的最低权限 |
| 环境隔离 | 不同环境使用不同权限级别的用户 |
| 密码策略 | 使用强密码，定期更换 |
| 访问限制 | 生产环境用户限制来源IP（'bemp_readonly'@'10.20.%'） |
| 审计追踪 | 启用MySQL通用日志或慢查询日志 |
| 权限回收 | 项目结束后及时回收临时授权 |

## 2. 数据保护

### 2.1 执行前备份

| 操作类型 | 备份要求 | Oracle 备份方式 | MySQL 备份方式 |
|----------|----------|----------------|---------------|
| DDL变更 | 必须备份表结构 | `describe_table` 记录 | `DESCRIBE` 记录 |
| DML变更（影响<100行） | 建议备份数据 | `execute_query` SELECT快照 | `execute_sql` SELECT快照 |
| DML变更（影响>=100行） | 必须备份数据 | SELECT快照+导出 | SELECT快照+导出 |
| 全表操作 | 必须全表备份 | `CREATE TABLE AS SELECT` | `CREATE TABLE ... AS SELECT` |

### 2.2 敏感数据处理

- 密码字段不得出现在查询结果中
- 客户账号需脱敏后才能导出
- 日志中不得记录敏感字段值
- 报告中敏感信息使用 `***` 替代

### 2.3 数据隔离

- 通过 LEGAL_NO 字段实现多法人数据隔离
- 查询时必须包含 LEGAL_NO 条件
- 跨法人操作需特殊审批

## 3. 操作安全

### 3.1 禁止操作

| 禁止项 | 原因 | 替代方案 |
|--------|------|----------|
| SELECT * | 性能和可维护性 | 明确列出字段 |
| 不带WHERE的DELETE/UPDATE | 误操作风险 | 必须加WHERE条件 |
| 生产环境直接操作 | 安全风险 | 通过审批流程执行 |
| 修改系统表 | 系统稳定性 | 通过标准接口修改 |
| 使用TRUNCATE | 不可回退 | 使用DELETE+COMMIT |

### 3.2 MySQL 特有禁止操作

| 禁止项 | 原因 | 替代方案 |
|--------|------|----------|
| 不带LIMIT的大数据量DELETE | 锁表风险 | 分批DELETE+LIMIT |
| DROP DATABASE | 不可恢复 | 逐表DROP |
| GRANT/REVOKE via MCP | 权限管理风险 | 通过DBA操作 |
| SET GLOBAL变量 | 影响全局 | 通过DBA操作 |

### 3.3 必须操作

| 必须项 | 原因 |
|--------|------|
| 执行前预检查 | 避免错误操作 |
| 执行前数据快照 | 支持回退 |
| 执行后验证 | 确保结果正确 |
| 生成执行报告 | 审计追溯 |

### 3.4 操作时间窗口

| 环境 | 允许操作时间 | 说明 |
|------|-------------|------|
| 开发 | 全天 | 无限制 |
| 测试 | 工作日 8:00-20:00 | 避免夜间操作 |
| 生产 | 维护窗口期 | 需提前申请 |

## 4. 异常处理

### 4.1 异常分类

| 异常类型 | 严重程度 | Oracle 处理方式 | MySQL 处理方式 |
|----------|----------|----------------|---------------|
| 连接失败 | 中 | 重试3次，仍失败则终止 | 重试3次，仍失败则终止 |
| SQL语法错误 | 高 | 停止执行，修正后重试 | 停止执行，修正后重试 |
| 约束冲突 | 高 | 停止执行，评估回退 | 停止执行，评估回退 |
| 执行超时 | 中 | 终止当前语句，评估影响 | 终止当前语句，评估影响 |
| 数据不一致 | 极高 | 立即回退，排查原因 | 立即回退，排查原因 |

### 4.2 MySQL 特有异常码

| 错误码 | 错误名称 | 说明 | 处理策略 |
|--------|---------|------|----------|
| 1062 | ER_DUP_ENTRY | 主键/唯一键冲突 | 检查数据，使用INSERT IGNORE或REPLACE |
| 1215 | ER_CANNOT_ADD_FOREIGN | 无法添加外键 | 检查关联表和字段类型 |
| 1451 | ER_ROW_IS_REFERENCED | 外键约束-被引用 | 先删除引用数据 |
| 1452 | ER_NO_REFERENCED_ROW | 外键约束-引用不存在 | 先创建被引用数据 |
| 1045 | ER_ACCESS_DENIED_ERROR | 访问被拒绝 | 检查用户名密码和权限 |
| 1049 | ER_BAD_DB_ERROR | 数据库不存在 | 检查数据库名称 |
| 1054 | ER_BAD_FIELD_ERROR | 字段不存在 | 检查字段名 |
| 1146 | ER_NO_SUCH_TABLE | 表不存在 | 需先执行DDL |
| 2006 | CR_SERVER_GONE_ERROR | 服务器连接断开 | 重连后重试 |
| 2013 | CR_SERVER_LOST | 查询期间连接丢失 | 重连后重试 |

### 4.3 异常上报

| 严重程度 | 上报对象 | 上报时限 |
|----------|----------|----------|
| 中 | 开发负责人 | 2小时内 |
| 高 | 开发负责人+DBA | 1小时内 |
| 极高 | 项目经理+DBA+架构师 | 立即 |

### 4.4 异常恢复

1. 确认异常影响范围
2. 执行回退操作恢复数据
3. 验证回退结果
4. 排查异常根因
5. 修正后重新执行
6. 记录异常处理过程

## 5. 审计合规

### 5.1 操作日志

所有数据库操作必须记录以下信息：

| 字段 | 说明 | 示例 |
|------|------|------|
| 操作时间 | 精确到秒 | 2026-05-16 14:30:25 |
| 数据库类型 | Oracle/MySQL | MySQL |
| 操作人员 | 执行人 | 张三 |
| 操作类型 | 操作分类 | DML-INSERT |
| 目标表 | 操作对象 | TM_AUTHORITY |
| SQL语句 | 执行的SQL | INSERT INTO ... |
| 影响行数 | 数据变更量 | 5 |
| 执行状态 | 成功/失败 | 成功 |
| 执行耗时 | 毫秒 | 125ms |

### 5.2 日志保留

- 开发环境：3个月
- 测试环境：6个月
- 生产环境：永久

### 5.3 合规检查

定期进行以下合规检查：

- [ ] 操作日志是否完整
- [ ] 审批流程是否合规
- [ ] 敏感数据是否脱敏
- [ ] 回退脚本是否配套
- [ ] 执行报告是否归档
- [ ] MySQL DDL/DML操作是否经过预检查
