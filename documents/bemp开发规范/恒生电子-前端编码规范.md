1 引言 
本文旨在为前端开发人员提供统一的编码规范，可以更快、更高效、更规范的完成繁重、
复杂、多样化的任务。制作规范的主要目的在于： 
• 降低每个组员介入项目的门槛成本； 
• 提高工作效率及协同开发的便捷性； 
• 高度统一的代码风格；

约束效力 
强制性

违反该项将被认为代码存在轻微缺陷 
根据具体产品特性的不同，选择性地遵守

4 编码规范 
4.1 HTML/TEMPLATE 开发规范 
4.1.1 通用语法 
1) 
【强制】缩进使用两个空格代替 Tab 
说明：前端代码层级较深，使用短缩进有利于利用屏幕空间，提升效率 
使用两个空格代替 Tab 可以保证在所有环境下获得一致展现 
<!--  not good--> 
<div> 
    <div>bar</div> 
</div> 
 
<!--  good --> 
<div> 
  <div>bar</div> 
</div> 
2) 
【强制】嵌套元素应当缩进一次（即两个空格），同层级缩进应保持一致 
<!--  not good--> 
<div> 
<div>bar</div> 
    <div>bar</div> 
</div> 
 
<!--  good --> 
<div> 
  <div>bar</div> 
  <div>bar</div> 
</div> 
3) 
【强制】将每个块元素、列表元素或表格元素都放在新行； 
•  inline 元素视情况换行，以长度不超过编辑器一屏为宜； 
•  每个子元素都需要相对其父级缩进； 
4) 【强制】对于属性的定义，使用双引号，不要使用单引号 
<!--  not good--> 
<input class='a' type=text> 
 
<!--  good --> 
<input class="a" type="text">

11) 【强制】不要在https 的链接里写http 的图片 
说明：只要https 的网页请求了一张http 的图片，就会导致浏览器地址栏左边的小锁
没有了，一般不要写死，写成根据当前域名的协议去加载，用//开头： 
<img src="//static.chimeroi.com/hello-world.jpg"> 
12) 【强制】所有元素必须正确嵌套,不允许交叉嵌套 
13) 【推荐】链接与锚点 
说明：在 a 标签中的 href，没有具体指向链接时，应该使用“javascript:void(0)”，而不
是“#”，避免导致页面不必要的跳转；需要在页内跳转时，请使用 #id 的方式 
14) 【推荐】不要在自闭合（self-closing）元素的尾部添加斜线 
说明：HTML 中一些元素是没有封闭标签的，应保持其用法， HTML5 规范中说明这是可
选的，元素如：area ,  base ,  br ,  col ,  command ,  embed ,  hr ,  img ,  input ,  
keygen , link ,  meta ,  param ,  source ,  track ,  wbr 
15) 【推荐】不使用属性设置样式（img, table 等元素） 
<!-- not good --> 
<img src="test.jpg" alt="test" width="400" height="200"> 
   
<!-- good --> 
<img src="test.jpg" alt="test" style="width: 400px; height: 200px;"> 
16) 【推荐】文件或类中的私有属性，变量和方法名应该以下划线 "_" 开头；保护属性，变
量和方法名不需要下划线开头，和公共变量名一样。建议结合 ES6 的私有属性规范，
用 # 作为类的私有属性修饰符。 
 
17) 【参考】自定义属性要以data-开头 自己添加的非标准的属性要以data-开头，否则w3c 
validator 会认为是不规范的 
<!—  not good—> 
<div count="5"></div>

<div data-count="5"></div>

4.1.3 语言属性 
1) 
【强制】为每个 HTML 页面根元素添加 lang 属性 
说明：根据 HTML5 规范： 强烈建议为 html 根元素指定 lang 属性，从而为文档设置正
确的语言。这将有助于语音合成工具确定其所应该采用的发音，有助于翻译工具确定其翻译
时所应遵守的规则等等。 
4.1.4 字符编码 
1) 
【强制】通过声明一个明确的字符编码，让浏览器轻松、快速的确定网页内容渲染方式，
通常指定为'UTF-8' 
<!DOCTYPE html> 
<html lang="zh-CN"> 
<head> 
  <meta charset="utf-8"> 
。。。 
2) 
【强制】标签书写 
说明：所有的 HTML 标签必须小写；所有的 HTML 属性必须小写； 
4.1.5 引入 CSS 和 JavaScript 文件 
1) 
【强制】根据 HTML5 规范，在引入 CSS 和 JavaScript 文件时不需要指定 type 属性，
因为 text/css 和 text/javascript 分别是它们的默认值 
2) 
【推荐】脚本加载放在body 之后 
说明：浏览器在解析 HTML 时，是按照标签进行顺序加载解析的，脚本引入如果放在 
head 中，会阻塞 DOM 的加载解析，在设计上脚本主要用于处理交互逻辑的，不应该影响
首次呈现过程，同时脚本文件的引入也应该放在 HTML 内容（body）的之后。 
建议将脚本的引入放在 body 尾部。 
3) 
【推荐】样式加载放在head 中 
说明：样式对 DOM 的渲染影响机制是先声明的优先生效，建议放在内容（body）之前
进行加载（head）。样式资源本身建议规划为基础样式、全局样式、UI 样式。

1) 
【推荐】编写 HTML 代码时，尽量避免多余的层级

1)

4.1.9 注释 
1) 
【推荐】尽可能的为代码写上注释，解释为什么要这样写，解决了什么问题等，在每个

2) 
【推荐】中英文之间、中文和数字之间用空格隔开。 
 
4.1.10 多媒体回溯 
1) 
【推荐】对于页面上的媒体而言，如图片、视频、canvas 动画，建议增加可替代的识别 
UI，如图片可以使用 alt 增加文字说明，alt 属性的内容为对该图片的简要描述，这对
于盲人用户和图像损毁都非常有意义，即无障碍。对于纯粹的装饰性图片，alt 属性值
可以留空，如 alt=""；视频和音频可以增加字幕和文字说明，能够真实描述内容而不是
用途。 
4.2 CSS/LESS/SASS 开发规范 
4.2.1 命名 
1) 
【强制】类名及id、关键帧命名可以使用大写字母、小写字母，允许以中划线(“-”) 或者
下划线(“_”)进行分隔 
// foo-bar 
// FooBar 
// foo-bar--foo-bar 
// foo-bar__foo-bar 
// FooBar--FooBar 
// FooBar__FooBar 
// FooBar--foo-bar 
// foo-bar--FooBar 
// Foo-bar 
2) 
【强制】不允许使用拼音，尤其是缩写的拼音、拼音与英文的混合 
3) 
【强制】命名-词汇规范 
•  不依据表现形式来命名； 
•  可根据内容来命名； 
•  可根据功能来命名。 
不推荐：left, right, center, red, black

4.5 前端请求调用规范 
请求调用是系统前端开发过程中较为常见的场景， 一般情况下，开发者会倾向基于现
有的三方库来实现请求调用，比如Vue.js 官方就推荐基于Axios 作为统一的请求调用库。但
是，三方库的引入会给系统开发带来一定的风险和不确定性，并且自身存在漏洞的三方库也
无法通过公司内部的依赖扫描。 
此外，三方库提供的接口可能过于宽泛，同样的功能基于不同的接口实现时会存在一定
的代码差异，在这种情况下，当我们需要对请求调用的过程进行统一的干预和处理时，比如
在全链路需求当中需要在请求调用时传入统一的trace_id 字段，不同的代码实现就需要分别
进行处理，不仅增加了开发工作量，也不便于研发过程的管理。 
本章节对基于HUI 框架进行前端开发的业务系统，约定了对应规范，用于规避上述提
到的各种问题。 
1) 
【强制】业务系统开发过程当中应该优先基于hCore 提供fetch 接口完成或封装业务请
求的调用方法。 
说明：应用框架hCore 基于Axios 封装了统一的请求调用接口，一方面对三方依赖
做了统一管理，便于进行维护和升级；另一方面，简化了请求配置，对请求的返回值和
异常信息做了统一处理，支持T2/T3 协议的应答适配，支持请求日志回溯；还基于中间
件技术实现了对请求调用的扩展，并提供了一系列业务通用的中间件。开发者可以专注
于业务的实现，在一定程度上提高了编码效率和代码质量，也便于后续的迭代升级。

【参考】业务系统应该封装和维护独立的请求调用方法，合理定义出入参，避免与业务
模块产生耦合。 
3) 
【参考】业务系统应该从权限校验的维度来构建fetch 实例。一般地，当不同请求的权
限校验逻辑一致时应该复用相同的fetch 实例。 
说明：请求调用的过程中往往要经过一定的权限校验，典型的比如会话权限的校验
和用户权限的校验等等。权限校验的逻辑往往是业务无关的，在不同的业务请求过程当
中可以复用。应用框架hCore 提供的fetch 接口支持多实例，同一个fetch 实例应该对应
一致的权限校验逻辑，便于统一维护。 
4) 
【推荐】业务系统应当基于请求中间件扩展fetch 实例的功能范围，比如扩展默认配置，
请求异常处理等。 
5) 
【推荐】业务系统请求中间件应当遵循职责单一原则，即一个中间件应当只做一件事情，
避免中间件的功能耦合。
