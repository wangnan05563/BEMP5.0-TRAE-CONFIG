引言 
本文旨在为 web 前端开发人员使用HUI提供一系列的编码规范，帮助我们更快、更好、
更高效的完成繁重、复杂、多样化的任务，我们制作规范的主要目的在于： 
• 降低每个组员介入项目的门槛成本； 
• 提高工作效率及协同开发的便捷性； 
• 高度统一的代码风格； 
2

约束等级 
约束效力 
强制性

违反该项将被认为代码存在严重缺陷 
前端程序团队必须遵守

违反该项将被认为代码存在轻微缺陷 
根据具体产品特性的不同，选择性地遵守

从产品持续优化及人员技能提升的角度，参考使用

4.1.1 通用语法 
1) 
【强制】缩进使用两个空格代替 Tab 
前端代码层级较深，使用短缩进有利于利用屏幕空间，提升效率 
使用两个空格代替 Tab 可以保证在所有环境下获得一致展现 
 
2) 
【强制】嵌套元素应当缩进一次（即两个空格），同层级缩进应保持一致 
 
3) 
【强制】将每个块元素、列表元素或表格元素都放在新行； 
•  inline 元素视情况换行，以长度不超过编辑器一屏为宜； 
•  每个子元素都需要相对其父级缩进； 
4) 
【强制】对于属性的定义，使用双引号，不要使用单引号

4.1.4 字符编码 
1) 
【强制】通过声明一个明确的字符编码，让浏览器轻松、快速的确定网页内容渲染方式，
通常指定为'UTF-8' 
 
2) 
【强制】标签书写 
所有的 HTML 标签必须小写；  所有的 HTML 属性必须小写； 
4.1.5 引入 CSS 和 JavaScript 文件 
1) 
【强制】根据 HTML5 规范，在引入 CSS 和 JavaScript 文件时不需要指定 type 属性，
因为 text/css 和 text/javascript 分别是它们的默认值 
 
2) 
【推荐】脚本加载放在body 之后 
浏览器在解析 HTML 时，是按照标签进行顺序加载解析的，脚本引入如果放在 head 中，
会阻塞 DOM 的加载解析，在设计上脚本主要用于处理交互逻辑的，不应该影响首次
呈现过程，同时脚本文件的引入也应该放在 HTML 内容（body）的之后。 
建议将脚本的引入放在 body 尾部。 
3) 
【推荐】样式加载放在head 中 
样式对 DOM 的渲染影响机制是先声明的优先生效，建议放在内容（body）之前进行
加载（head）。样式资源本身建议规划为基础样式、全局样式、UI 样式。

4.4.7.6  类型转换 
类型转换是大家常犯的错误，因为 JavaScript 是动态类型语言，不能指定变量的类型。 
把数字转换成字符串，应用"" + 1，虽然看起来比较丑一点，但事实上这个效率是最高的，
性能上来说： 
("" + ) > String() > .toString() > new String() 
这条其实和下面的“直接量”有点类似，尽量使用编译时就能使用的内部操作要比运行
时使用的用户操作要快。String()属于内部函数，所以速度很快，而.toString()要查询原型中的
函数，所以速度逊色一些，new String()用于返回一个精确的副本。  
浮点数转换成整型，不要使用 parseInt()，因为 parseInt()是用于将字符串转换成数字，
而不是浮点数和整型之间的转换。应该使用Math.floor()或者 Math.round()。 
另外，和对象查找中的问题不一样，Math 是内部对象，所以Math.floor()其实并没有多
少查询方法和调用的时间，速度是最快的。  
对于自定义的对象，如果定义了 toString()方法来进行类型转换的话，推荐显式调用 
toString()，因为内部的操作在尝试所有可能性之后，会尝试对象的 toString()方法尝试能否
转化为 String，所以直接调用这个方法效率会更高。 
4.4.7.7  使用直接量 
这个影响比较小。比如，JavaScript 支持使用[param,param,...]来直接表达一个数组，比
起使用 new Array(param,param,...)，前者是引擎直接解释的，后者要调用一个 Array 内部构
造器，所以要略微快一点点。同样，var foo = {};的方式也比 var foo = new Object();快，var reg 
= /../;要比 var reg = new RegExp()快。 
4.4.7.8  字符串遍历操作 
对字符串进行循环操作，譬如替换、查找，应使用正则表达式。因为本身JavaScript 的
循环速度就比较慢，而正则表达式的操作是用 C 写成的语言的API，性能很好。 
4.4.7.9  高级对象 
自定义高级对象和 Date、RegExp 对象在构造时都会消耗大量时间。如果可以复用，应
采用缓存的方式
